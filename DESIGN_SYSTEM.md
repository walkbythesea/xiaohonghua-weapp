# 小红花目标簿 · Design System

> 这份文档是项目的视觉与交互规范的唯一来源（single source of truth）。
> 所有 UI 改动都必须先读这份文档，再动代码。
>
> **代码层面的实现**在 `styles/tokens.wxss` 和 `styles/components.wxss` 中。
> 这份 Markdown 解释**为什么**这样设计，以及在边缘情况下应该怎么判断。
>
> 本文档的强制规则会在 `.cursorrules` 中以约束形式出现，AI 必须遵守。

## 一、设计哲学

小红花目标簿的视觉语言由三个原则定义。任何设计决策出现冲突时，按这三条排序判断。

### 原则 1：温暖克制（Warm Restraint）

产品是关于"陪伴用户日复一日的小坚持"，不是"催促用户完成任务"。所以视觉语言必须**温暖但不喧闹**。

具体翻译成规则：
- 主背景使用 warm beige（暖米色），不是冷白或纯白
- 阴影必须带暖色偏移（橙棕色相），**绝对不允许使用 `rgba(0, 0, 0, ...)`**
- 强调色（accent orange）整屏出现次数严格控制，不允许"全屏开花"
- 文字色不使用纯黑 `#000`，使用暖近黑 `#2A1F15`

### 原则 2：明确的视觉权重（Clear Visual Hierarchy）

每一屏只有一个"视觉中心"，这个中心必须是**用户当前最重要的任务**，不是产品方想推广的功能。

具体翻译成规则：
- 一屏内最饱和的强调色 `var(--accent)` 出现次数不得超过 2 处
- 次要功能（提醒开关、设置入口等）必须使用降级样式：暖灰底 + 深棕字，禁止使用强调色
- 主操作 / 次操作 / 三级操作必须有明确的视觉分层（见下文「交互权重规则」）

### 原则 3：细节即质感（Texture in Details）

廉价感和质感的差距往往不在大色块，而在阴影色温、边框粗细、字号节奏、留白比例这些"看不见"的细节上。

具体翻译成规则：
- 所有卡片必须有 `0.5px` 暖色 hairline border + 多层暖色阴影叠加
- 背景不使用纯色，使用 radial gradient + noise texture 增加层次
- 字号使用明确的 type scale，不允许"差不多大小"的随意字号
- 间距遵循 8 的倍数 scale，不允许 13px、19px 这种随意值

## 二、Design Tokens（视觉系统）

### 2.1 颜色 Colors

#### 背景色

| Token | Value | 用途 |
|---|---|---|
| `--bg-page` | `#F5EDE2` | 页面主背景。warm beige，比纯白多一层情绪温度。 |
| `--bg-card` | `#FFFFFF` | 主卡片背景。与页面背景形成柔和对比。 |
| `--bg-card-soft` | `#FBF6EE` | 次级卡片 / chip / 输入框背景。降级用。 |

**为什么不用纯白页面背景**：纯白 + 白卡片需要靠强阴影区分层次，会显得廉价。米色页面 + 白卡片用极淡阴影就能形成层次，且天然温暖。

#### 文字色

| Token | Value | 用途 |
|---|---|---|
| `--text-primary` | `#2A1F15` | 标题、重要文字。暖近黑，不是纯黑。 |
| `--text-secondary` | `#8B7355` | 正文、说明文字。暖棕灰。 |
| `--text-tertiary` | `#B8A48A` | 标签、次要信息、placeholder。 |

**判断规则**：如果一段文字超过一行且需要被阅读，用 primary。如果是辅助说明，用 secondary。如果用户可以完全跳过不看，用 tertiary。

#### 强调色（这是最关键的一组，使用必须克制）

| Token | Value | 用途 |
|---|---|---|
| `--accent` | `#E8632A` | **唯一的主强调色**。整屏最多 2 处。 |
| `--accent-soft` | `#FCE4D2` | 强调色的低饱和版本，用作 hot badge 的底色。 |
| `--accent-deep` | `#C44A15` | hover / pressed 状态。日常不直接使用。 |

**强调色使用清单（允许的场景）**：
- ✅ Hero 标题中需要被高亮的关键信息（如"今天还有 **2 个目标**"中的数字）
- ✅ 当前最活跃 streak 的 hot badge（带火苗图标）
- ✅ 主卡片上的"打卡"按钮文字
- ✅ Tab bar 当前选中项
- ❌ 不允许：所有卡片标签条、所有徽章、所有开关、所有箭头

**为什么这条这么严**：v1 设计的失败就是因为强调色到处用，结果用户视线无处停留。每一处强调色都是"花掉一次用户注意力预算"，预算用完了，整屏就没有重点。

#### 辅助色

| Token | Value | 用途 |
|---|---|---|
| `--line` | `rgba(139, 115, 85, 0.10)` | 所有 hairline border。warm taupe，不是冷灰。 |

### 2.2 阴影 Elevation（质感核心）

阴影是这套设计系统里**最重要**的细节。所有阴影必须满足两个条件：
1. **色相是暖色**（rgba(180, 120, 60, ...)），永远不用纯黑
2. **多层叠加**模拟真实光照（near + mid + ambient）

| Token | Value | 用途 |
|---|---|---|
| `--shadow-sm` | 双层极淡暖阴 | 微妙抬升，用于半透明 chip / pill |
| `--shadow-card` | 三层暖阴叠加 | 主卡片默认态。这是最常用的一个。 |
| `--shadow-card-pressed` | 单层紧贴阴 | 卡片按下态。卡片"压"向背景的物理感。 |

具体数值见 `styles/tokens.wxss`。

**为什么阴影这么重要**：人眼对阴影的感知是潜意识的——你说不上来为什么 Apple 的 UI 看起来"贵"，但其实就是阴影做对了。色温错了（用了纯黑），整张图立刻"塑料感"。

### 2.3 间距 Spacing

8 的倍数 scale：

| Token | px | rpx | 常见用途 |
|---|---|---|---|
| `--sp-1` | 4 | 8rpx | icon 与文字的微距 |
| `--sp-2` | 8 | 16rpx | 紧密元素的间距 |
| `--sp-3` | 12 | 24rpx | 卡片内段落间距 |
| `--sp-4` | 16 | 32rpx | 卡片内 padding 标准值 |
| `--sp-5` | 20 | 40rpx | 卡片之间的间距 / 卡片 padding 加大版 |
| `--sp-6` | 24 | 48rpx | section 之间的间距 |
| `--sp-7` | 32 | 64rpx | hero 区上下留白 |
| `--sp-8` | 40 | 80rpx | 大段视觉分隔 |

**铁律**：禁止使用 13rpx、19rpx、26rpx 这种非 scale 数值。如果某个具体场景非常需要 18rpx，停下来想想是不是别的地方设计错了，而不是新加一个数值。

### 2.4 圆角 Radius

| Token | Value | 用途 |
|---|---|---|
| `--r-sm` | 20rpx (10px) | 小元素：chip / 输入框 / 小按钮 |
| `--r-md` | 32rpx (16px) | **卡片标准圆角**。最常用。 |
| `--r-lg` | 40rpx (20px) | 大型容器 / 模态层 |
| `--r-pill` | 9999rpx | 完全圆角 pill。徽章、tab 高亮。 |

### 2.5 字体 Typography

字体族：`Noto Sans SC, -apple-system, sans-serif`

| 用途 | font-size | weight | line-height | letter-spacing |
|---|---|---|---|---|
| Hero 大标题 | 52rpx (26px) | 700 | 1.25 | -1rpx |
| Section 标题 | 34rpx (17px) | 600 | 1.4 | -0.4rpx |
| 卡片标题 | 32rpx (16px) | 600 | 1.4 | -0.4rpx |
| 正文 body | 26rpx (13px) | 400 | 1.5 | 0 |
| Caption | 22rpx (11px) | 500 | 1.4 | 0.6rpx |
| Section label (uppercase) | 22rpx (11px) | 500 | 1.4 | 2rpx |

**节奏铁律**：标题与正文的字号比必须 ≥ 1.3。如果太接近，层级感消失，整屏视觉变平。

## 三、组件规范

### 3.1 卡片 Card

**默认卡片** (`.ds-card`)
- 背景：`var(--bg-card)`
- 圆角：`var(--r-md)`
- Padding：`var(--sp-5)` 全方向
- Border：`1rpx solid var(--line)`
- Shadow：`var(--shadow-card)`
- 整张卡片可点击，按下时切换到 `var(--shadow-card-pressed)` 并微缩 0.985

**主卡片** (`.ds-card.ds-card-primary`)
- 在默认卡片基础上，背景改为 `linear-gradient(180deg, #FFFFFF 0%, #FFFAF3 100%)`
- 用于一屏中"最重要的那一张"——通常是 streak 最长的目标

**判断规则**：一屏只允许 0 或 1 张 primary 卡片，不允许多张。如果都重要，那就都用普通卡片。

### 3.2 徽章 Badge

**默认徽章** (`.ds-badge`)
- 字号：22rpx，font-weight 500
- 颜色：`var(--text-secondary)`
- 背景：`var(--bg-card-soft)`
- Padding：`10rpx 22rpx`
- 圆角：`var(--r-pill)`
- Border：`1rpx solid var(--line)`

**Hot 徽章** (`.ds-badge.ds-badge-hot`)
- 颜色改为 `var(--accent)`
- 背景改为 `var(--accent-soft)`
- font-weight 改为 600
- 必须搭配火苗图标使用

**判断规则**：一屏最多 1 个 hot 徽章。多个目标都"很活跃"时，只把 streak 最长的标记为 hot，其他用默认徽章。

### 3.3 按钮 Button

| 层级 | 视觉 | 用途 |
|---|---|---|
| Primary | 实心 accent 背景 + 白字 + 中等阴影 | 全屏唯一的主操作（如"创建新目标"） |
| Secondary | 透明背景 + accent 边框 + accent 字 | 次要操作 |
| Text | 无背景无边框 + accent 字 + 箭头 | 三级操作（如卡片内"打卡 →"） |

**铁律**：一屏只能有 1 个 Primary 按钮。如果出现两个 primary，停下来想想哪个才是真正的主操作。

### 3.4 列表行 List item

- 背景透明（继承父容器），不要给列表行加 `var(--bg-card)`
- 行间用 `1rpx solid var(--line)` 分隔，不用阴影
- Padding：上下 `var(--sp-4)`，左右继承父容器

### 3.5 开关 Switch

- 开启状态：背景 `var(--accent)`
- 关闭状态：背景 `rgba(139, 115, 85, 0.2)`
- **不要使用渐变或发光效果**，那会让开关看起来比卡片还重要

## 四、交互权重规则（直接解决"轻重不分"问题）

这一节是设计审查时最先检查的部分。

### 4.1 视觉中心规则

**每一屏必须有且只有一个视觉中心。**

视觉中心的定义：用户打开页面 0.5 秒内，视线第一个落到的元素。

如何判断哪个是中心：
- 它对当前用户最重要吗？（不是对产品方最重要）
- 它是否使用了页面中"最强"的视觉手段？（最大字号 / 最深背景 / 最饱和颜色 / 最明显阴影）
- 一屏内是否只有它使用了这种最强手段？

**反例（v1 失败案例）**：首页视觉中心是右上角的"明日打卡提醒开关"——一个次要功能。原因是它用了整屏最饱和的橙色 + 最显眼的圆形开关。修复方法：把它降级为半透明小条，让位给真正的主角——streak 最长的目标卡片。

### 4.2 操作分级

每个可点击元素必须明确属于以下三级之一：

| 级别 | 视觉特征 | 一屏数量上限 |
|---|---|---|
| Primary | accent 实心背景 / hot 徽章 / 大字号 | 1–2 处 |
| Secondary | 卡片整体可点击 + 暖白背景 | 不限 |
| Tertiary | 文字按钮 / 小图标 / 半透明背景 | 不限 |

**铁律**：如果你发现自己在用 Primary 级别去强调一个 Tertiary 级的功能，停下来重新思考产品优先级，而不是调样式。

### 4.3 留白即权重

一个元素周围的留白越大，它的视觉权重越高。所以：
- 主卡片上下间距 `var(--sp-5)`
- Hero 区上下间距 `var(--sp-7)`
- 次要区域（如提醒栏）紧贴上一个 section，间距 `var(--sp-3)`

不要"平均分配"留白，那会让所有元素权重相等。

## 五、图标系统

### 5.1 强制规则

- **绝对禁止使用 emoji** 作为 UI 图标。emoji 的视觉风格不可控，会破坏整体质感。
- 所有图标必须是 SVG line icon 风格
- 线宽统一 `stroke-width: 1.8`
- linecap 和 linejoin 统一 `round`
- 颜色使用 `currentColor`，不要 hardcode

### 5.2 图标来源

推荐从以下库挑选并直接内嵌 SVG（不要用图标字体）：
- Lucide Icons (https://lucide.dev) ← 首选，与本设计系统最契合
- Tabler Icons (https://tabler.io/icons)

### 5.3 已使用图标登记

新增图标时请追加到此处，避免重复引入风格不统一的版本：

| 名称 | 用途 | 来源 |
|---|---|---|
| flame | hot streak 徽章 | Lucide |
| bell | 提醒 | Lucide |
| home | tab bar 首页 | Lucide |
| user | tab bar 我的 | Lucide |
| arrow-right | 文字按钮箭头 | Lucide |

## 六、背景与纹理

页面背景**不允许使用纯色**。必须使用以下分层：

1. **基础渐变**：`radial-gradient(ellipse at top, #FAF2E5 0%, #F5EDE2 40%, #EFE5D6 100%)`
2. **暖色光晕层**：两个柔和的 radial gradient，模拟环境光
3. **噪点纹理层**：SVG noise filter，opacity 约 0.5，mix-blend-mode multiply

具体实现见 `styles/tokens.wxss` 中的 `page` 选择器及 `::before` / `::after` 伪元素。

**为什么这么麻烦**：纯色背景永远显得廉价。这是参考过的所有"看起来很贵"的 App（Finch、Streaks、Spark 等）共有的秘密——他们的背景都不是纯色。

> 微信小程序兼容性提醒：`mix-blend-mode` 在部分 Android 机型上支持不稳定。如果发现真机上纹理失效，可以接受降级——基础渐变 + 光晕层这两层在所有机型上都可用。

## 七、动效 Motion

**克制原则**：每一处动效都必须有明确的功能目的，不允许"为了动而动"。

| 场景 | 动效 | 时长 / easing |
|---|---|---|
| 卡片按下 | scale 0.985 + 阴影变浅 | 200ms ease |
| 页面切换 | 小程序原生过渡 | 不自定义 |
| 加载状态 | 骨架屏（不用 spinner） | — |
| 数字变化（如打卡次数+1） | 200ms 短暂高亮 | 200ms ease-out |

**禁止**：弹跳、旋转、3D 翻转、粒子效果。这些跟产品调性完全不符。

## 八、Don'ts 反例清单

按"绝对禁止"程度排序，越靠前越严重：

1. ❌ 使用 `rgba(0, 0, 0, ...)` 作为阴影颜色 → 用 `var(--shadow-*)` 或暖色 rgba
2. ❌ Hardcode 任何颜色值 → 必须引用 `var(--text-*)` / `var(--bg-*)` / `var(--accent*)`
3. ❌ Hardcode 任何 padding / margin / font-size / border-radius → 必须引用 token
4. ❌ 一屏出现 3+ 处 `var(--accent)` 强调色
5. ❌ 给次要功能（提醒、设置、帮助）使用强调色或大字号
6. ❌ 使用 emoji 作为 UI 图标
7. ❌ 卡片顶部加"装饰条"或"标签栏"（v1 那种橙色渐变线）
8. ❌ 使用纯色背景，不加渐变和纹理
9. ❌ 字号"差不多大小"——必须严格遵循 type scale
10. ❌ 使用非 8 倍数的间距（13rpx、17rpx、19rpx 等）
11. ❌ 给开关、徽章、箭头加发光、渐变、阴影等额外效果
12. ❌ 一屏内出现两个 Primary 按钮

## 九、给 AI 的使用说明

如果你（AI）在读这份文档以辅助修改 UI 代码，请按以下流程：

### Step 1：先理解，再动手
- 阅读「一、设计哲学」「四、交互权重规则」「八、Don'ts 反例清单」
- 不要直接跳到 token 表然后开始替换——理解"为什么"才能在边缘情况下做对决策

### Step 2：参考已有实现
- `styles/tokens.wxss` 是所有视觉变量的实现
- `styles/components.wxss` 是已封装的通用组件类（`.ds-card`、`.ds-badge` 等）
- 优先复用 `.ds-*` 类，不要重新发明

### Step 3：写代码时
- 任何颜色、阴影、间距、字号、圆角必须引用 `var(--*)` 变量
- 任何卡片、徽章、按钮优先用 `.ds-*` 类
- 任何图标使用 SVG，禁用 emoji

### Step 4：写完后自检
逐条核对「八、Don'ts 反例清单」12 条。如果有任何一条违反，回去修改。

### Step 5：边缘情况
如果某个具体场景在这份文档中找不到答案，**不要自己发明新规则**。停下来询问用户，或者明确标注 `[TODO: 需要确认]` 让用户来定。新规则定下来后，要更新这份文档。
